# UI-Editor-kit v0.2.0

**Release-Typ:** interner stabiler Core-Fixstand

## Zusammenfassung

UI-Editor-kit v0.2.0 ist ein stabiler interner Fixstand des oeffentlichen Core. Der Stand buendelt Runtime-Start, offiziellen Ziel-App-Adapter-Pfad, ViewModels, LayoutState-Vertrag und MemoryLayoutStateStore ueber den oeffentlichen CommonJS-Einstieg `src/index.cjs` und den Paket-Export.

Es erfolgt keine npm-Veroeffentlichung. Dieses Release dient als gepruefte Basis fuer einen lokalen Git-Tag `v0.2.0` und ein anschliessendes GitHub-Release.

## Wichtigste Funktionen

- Oeffentlicher CommonJS-Paketimport ueber `src/index.cjs`.
- Stabiler Adapter-Pfad fuer neue fachneutrale Ziel-Apps.
- Runtime-Launcher fuer den kontrollierten Start aus HostAdapter, AdapterManifest, Registry und LayoutStateStore.
- Scope-, Selection-, Status- und Layout-Control-ViewModels fuer neutrale Editor-Zustaende.
- LayoutState-Vertrag inklusive Profilkompatibilitaet.
- MemoryLayoutStateStore als speicherinterne Referenzimplementierung.
- Neutrale Testziel-Integration und ausfuehrbares Minimalbeispiel.
- Oeffentliche Minimalanleitung fuer neue Ziel-Apps.

## Oeffentliche Core-API

Der oeffentliche Einstieg ist:

```js
const uiEditorKit = require("ui-editor-kit");
```

Alternativ kann im Repository direkt geladen werden:

```js
const uiEditorKit = require("./src/index.cjs");
```

Der Paket-Einstieg stellt unter anderem bereit:

- `validateTargetAppAdapterPath`
- `createTargetAppAdapterRuntime`
- `getTargetAppAdapterPathSummary`
- `createEditorRuntimeLauncher`
- `createEditorRuntimeStatusViewModel`
- `createEditorSelectionViewModel`
- `createEditorScopeViewModel`
- `createEditorLayoutControlViewModel`
- `validateLayoutState`
- `normalizeLayoutState`
- `createLayoutState`
- `getLayoutStateProfileKey`
- `assertCompatibleLayoutProfile`
- `createMemoryLayoutStateStore`

## Offizieller Ziel-App-Adapter-Pfad

Der offizielle Pfad fuer neue Ziel-Apps lautet:

```text
Target-App -> AdapterManifest -> HostAdapter -> Registry -> RuntimeLauncher -> ViewModels -> LayoutStateStore
```

Ziel-Apps liefern die notwendigen Adapter- und Registry-Daten explizit. Das Kit liest diese Daten, prueft den Adapter-Pfad und erzeugt daraus Runtime- und ViewModel-Zustaende.

## Runtime-Launcher

Der Runtime-Launcher startet fachneutral aus HostAdapter, optionalem AdapterManifest, Registry und LayoutStateStore. Er erzeugt kontrollierte Runtime-Zustaende und veraendert keine Ziel-App automatisch.

## Scope-, Selection-, Status- und Layout-Control-ViewModels

v0.2.0 enthaelt ViewModels fuer:

- Scope-Anzeige und Scope-Beschreibung,
- Auswahlzustand,
- Runtime-Status,
- Layout-Control-Aktionen fuer Save, Load und Reset.

Diese ViewModels sind neutral und enthalten keine Ziel-App-Fachlogik.

## LayoutState-Vertrag

Der LayoutState-Vertrag definiert die Struktur, Normalisierung und Profilkompatibilitaet fuer Layout-Zustaende. Save, Load und Reset bleiben an diesen Vertrag und kompatible Stores gebunden.

## MemoryLayoutStateStore

`createMemoryLayoutStateStore` stellt eine speicherinterne Store-Referenz bereit. Sie eignet sich fuer Tests, Minimalintegration und kontrollierte Integrationsbeispiele ohne externe Persistenz.

## Neutrale Testziel-Integration

Die neutrale Testziel-Integration prueft den Adapter-Pfad ohne konkrete Ziel-App-Abhaengigkeit. Das Minimalbeispiel zeigt die erwartete Anbindung ueber den oeffentlichen Einstieg.

## Oeffentliche Minimalanleitung

Die Minimalanleitung fuer neue Ziel-Apps steht in `docs/M47_NEUE_ZIEL_APP_MINIMAL_ANBINDUNG.md`. Die oeffentliche Core-API ist in `docs/M48_PUBLIC_CORE_API_EXPORTS.md` beschrieben.

## Kompatibilitaet

- CommonJS-Einstieg: `src/index.cjs`
- Paketfelder: `main` und `exports` zeigen auf den oeffentlichen Einstieg.
- Version in `package.json`: `0.2.0`
- Keine npm-Veroeffentlichung in diesem Release.

## Bekannte Grenzen

- Keine automatische Analyse bestehender Oberflaechen.
- Kein DOM-Scan.
- Keine automatische UI-Erkennung.
- Keine automatische Registry-Befuellung.
- Keine echte externe Persistenz.
- Keine konkrete Ziel-App-Integration innerhalb dieses Releases.

## Nicht-Ziele

Dieses Release erstellt keinen Git-Tag, veroeffentlicht kein GitHub-Release und fuehrt keine npm-Veroeffentlichung durch. Es fuegt keine neue Core-Funktion, Runtime-Logik, Adapter-Logik, UI, Electron-App, Datenbank oder Dateiablage hinzu.
